# Brief — claim orientation specialist

Read-only assembler. Given a claim_id, produces a one-screen
orientation for an adjuster who just opened the claim. Pulls from what
other specialists and services have already computed; does not
re-derive anything.

## What Brief does

**Input:** `Caseload` + `claim_id`.

**Output:** a `ClaimBrief` (existing schema in
`argos/schemas/specialists/brief.py`). Brief populates the fields it
can derive today and leaves the rest at safe defaults:

| Field | Source | Behavior when source missing |
|---|---|---|
| `story_paragraph` | LLM call over policy + loss facts + claim metadata | Always populated (one of the two LLM calls) |
| `story_citations` | Doc IDs cited in narrative | Min one citation required by schema |
| `current_status_snapshot` | Caseload flags + Coverage result if present | "pending" / defaults when specialist hasn't run |
| `missing_info` | Rule-based gap detection + LLM rationale lines | List can be empty |
| `since_last_touch` | — | `last_touch_at=None`, empty diff list (Changelog feature is separate) |
| `financial_snapshot` | — | Zeros + current timestamp (Reserve specialist owns this) |
| `workflow_recommendations_summary` | Persisted specialist results | Empty list when no specialists have run |
| `pending_communications` | — | Empty list (correspondence system not built) |

## What Brief does NOT do

- Does not re-analyze coverage, liability, reserve, recovery. It reads
  their results if files exist; renders "not yet analyzed" if not.
- Does not log "last touched by adjuster" or diff state across
  visits. That's the Changelog feature, separate spec.
- Does not act on `missing_info`. No fetching, no carrier emails, no
  orders. The gap list is for the adjuster to act on.
- Does not rank or prioritize claims. That's Triage.

## The two LLM calls

**Call 1 — narrative (`story_paragraph` + `story_citations`).**
One paragraph (≤ 4 sentences) covering: line of business, loss event
(what/when/where), named insured, claimant, severity tier, current
litigation/rep posture. Tool-use shape mirrors the Document Reader's
proven pattern. Citations point to the documents that ground the
narrative facts.

**Call 2 — gap rationales (`missing_info[].evidence_citations` +
rationale text).** Batched: one call covers all detected gaps. Input
is the list of {variable, status} detected by the rule layer. Output
is the {requested_from, why_it_matters one-liner, supporting
citations}. Citation is what makes the AI conclude the info is
missing — usually the absence of a doc type expected for the claim's
line of business.

## Rule-driven gap detection (deterministic, pre-LLM)

For v1 the gap list is generated by simple rules over Caseload state:

- Policy declarations doc missing → "Coverage limits unverified"
- No ISO claim search result → "Duplicate-claim risk unscreened"
- Severity ≥ 3 + no medical records → "Damages exposure unverified"
- Litigation flag + no attorney rep doc → "Plaintiff counsel unknown"
- Coverage specialist not yet run + claim age > 7d → "Coverage posture
  not yet analyzed"

Rules are a `dict[str, GapRule]` in `gaps.py`. Adding a rule is one
function and one entry; LLM only fills the rationale line.

## Orchestrator integration

Brief is registered in `WORKFLOW_REGISTRY` like Coverage. Unlike
Coverage, it is **not** dispatched by Reader (it is not
posture-triggered). The runner invokes it when the UI or a script asks
for an orientation on a specific claim.

Concretely:

- `runner.WORKFLOW_REGISTRY["brief"] = _run_brief_via_adapter`
- `dispatcher.POSTURE_TO_WORKFLOWS` is untouched.
- A new entry-point (`scripts/run_brief.py` or similar) creates a Job
  manually and feeds it to the runner.

## File layout

```

  __init__.py
  brief.py            # run_brief(caseload, claim_id) -> ClaimBrief
  assembler.py        # read-only assembly from caseload + results dir
  narrative.py        # LLM call for story_paragraph
  gaps.py             # rule-based gap detection + LLM call for rationales
tests/specialists/brief/
  test_assembler.py   # deterministic; no LLM
  test_gaps.py        # rule-based detection only; no LLM

scripts/run_brief.py  # one-claim invocation for the demo
```

## Eval — pre-locked thresholds

Three hand-written golden Briefs covering the three populated realistic
claims (REQ-007, REQ-013, REQ-015 — the ones with pinned realistic
docs). Each golden has:

- Expected narrative facts (LOB, parties, loss event, severity)
- Expected status snapshot (Coverage status if Coverage has run on the
  claim; "pending" otherwise)
- Expected gaps list

LLM-judge rubric:

- Narrative accuracy (no hallucinated facts) — pass = 0 hallucinations
  across 3 briefs
- Posture status correctly pulled when source exists, correctly
  "pending" when not — pass = 3/3 match
- Material activity matches Reader output verbatim — pass = exact set
  match
- Gap list overlap with golden — pass = recall ≥ 0.8

All four thresholds locked in `docs/evals/brief-locked-thresholds.md`
**before** the first eval run. Pre-registered: spec → lock → build →
one run → verdict.

## Build order

1. Spec doc (this file).
2. Assembler + deterministic gap rules + tests. No LLM. Existing
   schema covers the output shape — just populate it.
3. Narrative LLM call + tests against pinned docs.
4. Gap rationale LLM call + tests.
5. `run_brief.py` entry-point + register in orchestrator.
6. Locked eval doc + one eval run + verdict.
