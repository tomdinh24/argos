## Failure root causes

### C3 / CLM-007 — lock was wrong

The pre-locked expectation for CLM-007 was
`workflow_recommendations = empty`. In reality, `reserve.json`
(from the orchestrator demo's stub) was already on disk. The Brief
faithfully consumed it. **Brief behaved correctly; the lock didn't
account for stub-result persistence.**

### C4 / all three claims — check method too brittle

Locked check: `{m.item for m in brief.missing_info} == humanize_variable(rule.variable)` (exact-string).

Observed: the LLM produced **better** item names than the naive
humanizer:

| Rule variable | `humanize_variable()` | LLM output (various claims) |
|---|---|---|
| `iso_claim_search` | "Iso claim search" | "ISO claim search result" / "ISO ClaimSearch result" |
| `policy_declarations` | "Policy declarations" | "Policy declarations page" |
| `medical_records` | "Medical records" | "Claimant medical records" / "Complete medical records (pre- and post-accident)" |
| `coverage_analysis` | "Coverage analysis" | "Coverage analysis memo" / "Coverage analysis memorandum" |

**Brief behaved correctly; the eval check matches strings instead of
checking semantic recall.** Every rule-detected variable is covered by
the LLM output — just under a more natural name.

## Proposed revisions for run 2

1. **C3 lock revision** — expected set for CLM-007 = `{reserve}`,
   for CLM-013 = `{}`, for CLM-015 = `{coverage}`. Reflects what the
   orchestrator demo actually persists.
2. **C4 check method revision** — check that each rule-detected
   variable (e.g., `iso_claim_search`) appears as a case-insensitive
   substring or token-overlap match against `{m.item for m in
   missing_info}`. The pre-registration discipline still applies:
   variable set is locked, only the matching predicate is loosened
   from exact-match to substring/token-overlap.

Both revisions require a new locked thresholds doc with a `revised:`
line before run 2.
